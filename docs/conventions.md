# Conventions

This document defines code layout, import rules, and the per-module development workflow. Following these conventions is mandatory; the project is structured to make violations physically inconvenient.

## Working directory

All backend code lives under `backend/`. Unless stated otherwise, run every command in this document from the `backend/` directory:

```bash
cd backend
uv sync
```

The `docs/` folder lives at the repository root. The `template-fe/` folder is the legacy prototype; it is read-only and will be deleted per `decisions.md` D18 once migration is complete.

## Source layout

Every module under `backend/packages/` uses the **src layout**:

```
backend/packages/<module>/
├── pyproject.toml
└── src/
    └── <module>/
        ├── __init__.py
        └── *.py
```

Reasoning: tests can only import the module after it is installed via `uv sync`. This guarantees that what you test is what consumers will install. Flat layouts hide packaging bugs until first deploy.

Tests, scripts, and module-specific docs live OUTSIDE `src/`:

```
backend/packages/<module>/
├── pyproject.toml
├── README.md          # optional, module-specific
├── tests/
└── src/<module>/
```

## Cross-module imports

A module may only import from packages it has declared in its `pyproject.toml` `dependencies`. If `api` does not declare `embedding` as a dependency, then `from embedding.embedder import Embedder` will fail with `ModuleNotFoundError`.

| From → To | Allowed? |
|-----------|----------|
| Any module → `core` | Always (declared by every module) |
| `api` → ML modules (embedding, labeling, etc.) | Forbidden — api is intentionally lean |
| `pipeline` → batch modules | Yes — pipeline is the orchestrator |
| Batch module A → batch module B | Forbidden — each batch module is self-contained |

If you need to share logic between two batch modules, lift it into `core`. If `core` is becoming a junk drawer, that is a signal to split it — open a discussion before adding generic utility files.

## Module internal structure

Each batch module follows the same shape:

```
backend/packages/<module>/src/<module>/
├── __init__.py        # public API of the module
├── pipeline.py        # the entry function: run_daily(), run_for_date(), etc.
├── cli.py             # Click CLI exposing pipeline functions
└── *.py               # internal helpers
```

The `api` module follows FastAPI convention:

```
backend/packages/api/src/api/
├── __init__.py
├── main.py            # FastAPI app instance
├── deps.py            # FastAPI dependencies (DB session injection, etc.)
└── routes/
    ├── __init__.py
    ├── clusters.py
    ├── articles.py
    ├── sources.py
    └── health.py
```

The `core` module is organized by concern, not feature:

```
backend/packages/core/src/core/
├── __init__.py
├── config.py          # pydantic Settings
├── db.py              # async engine + session factory
├── models.py          # SQLAlchemy ORM (mirror of schema.dbml)
└── schemas.py         # shared Pydantic schemas (if any)
```

## Naming

| Element | Convention | Example |
|---------|-----------|---------|
| File | snake_case | `cluster_insight.py` |
| Class | PascalCase | `ClusterInsight` |
| Function / method | snake_case | `compute_velocity()` |
| Constant | UPPER_SNAKE | `DEFAULT_BATCH_SIZE` |
| Module | snake_case | `clustering`, not `Clustering` |
| Database table | snake_case singular | `article_cluster`, not `article_clusters` |
| Database column | snake_case | `published_at` |

## Testing

- Tests live in `backend/packages/<module>/tests/`.
- Use `pytest`. Async tests use `pytest-asyncio`.
- One test file per source file: `src/<module>/foo.py` ↔ `tests/test_foo.py`.
- DB-touching tests use a transactional fixture that rolls back after each test.
- Do not test ML model output quality in unit tests. Validate plumbing only (e.g., shape, types, persistence).

## Module development order

Modules must be developed bottom-up by dependency:

1. `core` — required by all others
2. `ingest` — produces articles for downstream
3. `embedding` — depends on articles existing
4. `clustering` — depends on embeddings
5. `labeling` — depends on clusters
6. `scoring` — depends on clusters
7. `api` — reads everything
8. `pipeline` — orchestrates 2-6

Do not start a module until its upstream is functional and tested. Each module is "done" when it can run end-to-end against real data persisted by upstream modules.

## Alembic workflow

Schema is sourced from `backend/packages/core/src/core/models.py` (SQLAlchemy ORM). Migrations are autogenerated and committed. Run from `backend/`:

```bash
# After editing models.py
alembic revision --autogenerate -m "add foo column"

# Review the generated SQL by hand. Autogenerate is imperfect — it can miss
# enum changes, custom types, and constraints. Adjust the migration file if needed.

# Apply
alembic upgrade head
```

Never edit `schema.dbml` directly to add new fields during development. `schema.dbml` documents the schema; `models.py` is the source of truth that Alembic reads.

## uv command cheat sheet

```bash
# Install workspace dependencies
uv sync

# Add a runtime dependency to a specific module
uv add --package ingest httpx

# Add a dev-only dependency to the workspace
uv add --dev pytest-mock

# Run a CLI defined in a module
uv run python -m ingest.cli run-rss

# Run tests for one module
uv run pytest packages/ingest/tests/

# Run the API locally
uv run uvicorn api.main:app --reload --app-dir packages/api/src

# Run the full daily pipeline
uv run python -m pipeline.cli run-daily
```

## Datetimes

All datetime values are **naive UTC** — `TIMESTAMP WITHOUT TIME ZONE` in Postgres. Never insert an aware datetime; strip the timezone with `.replace(tzinfo=None)` before persisting:

```python
from datetime import UTC, datetime

now = datetime.now(UTC).replace(tzinfo=None)
```

Do not use `datetime.utcnow()` — deprecated since Python 3.12.

## Code style

- Run `ruff check .` and `ruff format .` before committing. The project's root `pyproject.toml` configures ruff.
- Type hints are encouraged everywhere. Run `mypy` periodically; do not block on full type coverage in MVP.
- No comments explaining WHAT the code does. Identifier names are the documentation. Comments are reserved for non-obvious WHY: a hidden constraint, a workaround for a specific bug, or behavior that would surprise a reader.
- No multi-line docstrings. One-line module / class docstrings are fine when they add value.
- Follow the brevity guidance in the workspace-level `CLAUDE.md` if present.

## Commit style

- One logical change per commit.
- Commit message: short imperative subject ("add embedding singleton loader"), optional body for context.
- Do not commit secrets, credentials, or `.env` files.
- Do not commit downloaded model weights — they live in `~/.cache/huggingface` (mounted volume in Docker dev).

## Before introducing anything new

If you find yourself wanting to do any of the following, stop and consult `constraints.md` first:

- Add a new top-level module to the workspace
- Add a new runtime dependency
- Introduce a message queue, cache, or background worker pattern
- Add a write-side API endpoint beyond source management (see `constraints.md`)
- Add an authentication or user management feature
- Add monitoring or observability infrastructure
- Add a write-side API endpoint
